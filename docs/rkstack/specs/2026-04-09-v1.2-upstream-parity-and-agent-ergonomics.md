# Confluence MCP Plugin v1.2 — Upstream Parity Completion and Agent Ergonomics

**Date:** 2026-04-09
**Status:** Draft
**Branch:** main (v1.1.1 baseline)
**Context:** ChatGPT ran a real-world end-to-end test of the plugin and provided detailed feedback. This spec addresses the gaps identified.

## Problem

The plugin is functional and has good Markdown-to-Confluence conversion, but real-world agent usage revealed several friction points:

1. **Diff is unusable for content review** — shows raw storage format with noisy macro-ids and task-ids, making it impossible for an AI agent to understand what actually changed in the page content.
2. **Full-page replacement on update is dangerous** — agents must read the entire page, reconstruct it, and write it back. No protection against concurrent edits. No way to do targeted changes.
3. **No space discovery** — agents cannot list available Confluence spaces, forcing them to guess or extract space keys from search results.
4. **Response format inconsistencies** — `create_page` returns `{"page": {...}}` with storage format content, `get_page` returns `{"metadata": {...}}` with markdown. Different keys, different content formats.
5. **Page URL resolution missing** — agents often receive URLs from users but tools only accept page IDs. No way to resolve a URL to a page.

## Solution

Seven improvements organized into three tiers by implementation effort and impact.

### Tier 1: Quick Wins (low effort, high impact)

#### 1.1 Markdown-based diff

**Current:** `get_page_diff` fetches both versions as `body.storage.value` and diffs the raw XHTML. This produces noisy output full of `ac:macro-id`, `ac:task-id`, and other internal Confluence identifiers.

**Change:** Convert both versions to Markdown via `StorageToMarkdown.convert()` before computing the diff. The diff then shows what a human would see: content changes, not internal XML shuffling.

**Implementation:**
- In `GetPageDiffTool.execute()`, after fetching both versions' `body.storage.value`:
  - Convert each to Markdown: `StorageToMarkdown.convert(fromContent)` / `StorageToMarkdown.convert(toContent)`
  - Diff the Markdown strings instead of the raw storage
- Keep the JSON response format: `{page_id, from_version, to_version, diff}`
- No new parameters needed — markdown diff is strictly better for all agent use cases

#### 1.2 Unified response keys

**Current:** `create_page` and `update_page` return `{"message": "...", "page": {...}}`, while `get_page` returns `{"metadata": {...}}`. The content within uses different keys.

**Change:** Unify to `page` everywhere:
- `get_page` → `{"page": {...}}` (was `metadata`)
- `get_page_history` → `{"page": {...}}`
- `create_page`, `update_page`, `move_page` → keep `{"message": "...", "page": {...}}`
- `delete_page` → keep `{"success": true, "message": "..."}`

**Implementation:**
- Change `GetPageTool` to use `"page"` key instead of `"metadata"`
- Change `GetPageHistoryTool` similarly
- Update e2e test assertions that check for `"metadata"` key

#### 1.3 Add `list_spaces` tool

**Current:** No way for an agent to discover available Confluence spaces. Agents must guess space keys or extract them from search results.

**Change:** Add a new `list_spaces` tool that returns simplified space information.

**Implementation:**
- New class `tools/spaces/ListSpacesTool.java`
- API: `GET /rest/api/space?limit={limit}&start={start}&type={type}`
- Parameters: `limit` (default 25), `start` (default 0), `type` (optional: "global", "personal")
- Response: flat list of `[{key, name, type, url}]` matching upstream's simplified format
- URL constructed as `{baseUrl}/spaces/{key}/overview`
- Register in `ToolRegistry`
- Update `ALL_UPSTREAM_TOOLS` in e2e test

#### 1.4 Accept URL in `page_id` parameter

**Current:** All page tools require a numeric page ID. Agents often receive Confluence URLs from users.

**Change:** If `page_id` contains a URL, extract the page ID from it automatically.

**Implementation:**
- Add a static helper method `extractPageId(String input)` to a shared utility (or to `ResponseTransformer`)
- URL patterns to handle:
  - `https://confluence.example.com/pages/viewpage.action?pageId=123456` → `123456`
  - `https://confluence.example.com/spaces/KEY/pages/123456/Title` → `123456`
  - `https://confluence.example.com/display/KEY/Title` → requires API lookup by title+space (or reject with helpful error)
  - Plain numeric string → pass through as-is
- Apply in all tools that accept `page_id`: `GetPageTool`, `UpdatePageTool`, `DeletePageTool`, `MovePageTool`, `GetPageChildrenTool`, `GetPageHistoryTool`, `GetPageDiffTool`, `GetCommentsTool`, `AddCommentTool`, `GetLabelsTool`, `AddLabelTool`, `GetPageViewsTool`, and attachment tools
- Best approach: add the extraction to `McpTool` as a default method, or create a shared base class

### Tier 2: Safety and Usability (medium effort, high impact)

#### 2.1 Optimistic locking on `update_page`

**Current:** `update_page` always succeeds even if the page was modified between the agent's `get_page` and `update_page` calls. Silent data loss.

**Change:** Add optional `expected_version` parameter. When provided, the tool checks if the page's current version matches before updating. If it doesn't match, the update is rejected with a clear error message telling the agent to re-read the page.

**Implementation:**
- Add `expected_version` parameter to `UpdatePageTool.inputSchema()`
- Description: "If provided, the update will fail if the page's current version doesn't match this value. Use the version number from get_page to prevent overwriting concurrent changes."
- In `execute()`, after fetching current version: if `expected_version` is set and doesn't match `currentVersion`, throw `McpToolException("Page was modified since you last read it (current version: " + currentVersion + ", expected: " + expectedVersion + "). Re-read the page with get_page before updating.")`
- This is backward compatible — existing calls without `expected_version` continue to work

#### 2.2 Return markdown content from write tools

**Current:** `create_page` and `update_page` return the page content in storage format (`format: "storage"`). This means after creating a page, the agent must call `get_page` separately to see the markdown version.

**Change:** Add optional `return_markdown` parameter (default: false). When true, convert the response content from storage to markdown before returning.

**Implementation:**
- Add `return_markdown` parameter to `CreatePageTool` and `UpdatePageTool` inputSchema
- In `execute()`, after getting the API response, if `return_markdown` is true:
  - Extract `body.storage.value` from the raw response
  - Convert via `StorageToMarkdown.convert()`
  - Set content in simplified response as `{value: markdown, format: "markdown"}`
- Default is false to keep backward compatibility and avoid unnecessary conversion overhead

### Tier 3: Advanced Agent Primitives (higher effort, transformative impact)

#### 2.3 Partial page update tools

**Current:** `update_page` replaces the entire page content. For large pages, this is:
- Token-expensive (agent must read and reproduce the full page)
- Error-prone (agent may accidentally drop sections)
- Unsafe without optimistic locking

**Change:** Add two new tools for common partial update patterns:

**`append_to_page`** — adds content to the end of a page without touching existing content.
- Parameters: `page_id`, `content`, `content_format` (default: markdown)
- Implementation: fetch current page, concatenate new content to existing `body.storage.value`, update with incremented version
- The new content goes through `MarkdownToStorage.convert()` if format is markdown
- Use case: adding meeting notes, appending status updates, adding new sections

**`prepend_to_page`** — adds content to the beginning of a page.
- Same as `append_to_page` but inserts before existing content
- Use case: adding changelog entries, alerts, or announcements at the top

Both tools inherit optimistic locking behavior from 2.1 (optional `expected_version`).

**Why not `replace_section`?** Section-level replacement requires parsing the page content to find sections by heading, which is fragile across storage format variations. Append/prepend cover the most common business use cases (adding notes, updates, new entries) without parsing complexity. Section replacement can be added later if needed.

#### 2.4 Preview/dry-run mode

**Problem:** Agents cannot preview how Markdown will render in Confluence before committing. Mistakes require another update cycle.

**Change:** Add a `convert_only` tool that converts Markdown to Confluence storage format and returns the result without creating or updating any page.

**Implementation:**
- New class `tools/pages/ConvertContentTool.java`
- Parameters: `content` (markdown string), `content_format` (default: markdown)
- Response: `{storage_format: "...", preview_html: "..."}` — the converted storage format XHTML
- No page ID, no space key, no API call — pure conversion
- Useful for debugging, previewing, and validating content before write
- Also serves as a learning tool: agents can see what their Markdown produces

#### 2.5 Section-level page editing

**Problem:** Agents editing large pages must read and rewrite the entire page. This is token-expensive, error-prone, and risks losing content.

**Change:** Add `replace_section` tool that replaces content under a specific heading without touching the rest of the page.

**Implementation:**
- New class `tools/pages/ReplaceSectionTool.java`
- Parameters: `page_id`, `heading` (the heading text to find), `content` (new content for that section), `content_format` (default: markdown), `expected_version` (optional)
- Behavior:
  1. Fetch current page content as storage format
  2. Convert to markdown via `StorageToMarkdown`
  3. Find the section by heading (match `## Heading` or `### Heading`)
  4. Replace everything from that heading to the next heading of equal or higher level
  5. Convert back to storage via `MarkdownToStorage`
  6. Update the page with incremented version
- Edge cases: heading not found → error with available headings listed. Multiple matches → error.
- Optional `heading_level` parameter to disambiguate (e.g., `heading_level=2` for `##`)

#### 2.6 Batch page creation (page + labels + attachments)

**Problem:** Creating a fully-formed page requires multiple sequential tool calls: `create_page` → `add_label` (N times) → `upload_attachment` (N times) → `add_comment`. This is slow and fragile.

**Change:** Extend `create_page` with optional `labels` and `initial_comment` parameters.

**Implementation:**
- Add parameters to `CreatePageTool.inputSchema()`:
  - `labels` (array of strings) — labels to add after creation. Description: "List of labels to apply to the page after creation. Example: ['policy', 'hr', 'q3-2026']"
  - `initial_comment` (string) — a comment to add after creation. Description: "Optional comment to add to the page (in Markdown format)"
- In `execute()`, after successful page creation:
  1. If `labels` provided: POST each label to `/rest/api/content/{id}/label`
  2. If `initial_comment` provided: POST comment to `/rest/api/content` with container reference
  3. Include label and comment results in response: `{"message": "...", "page": {...}, "labels_added": [...], "comment_added": true}`
- Failure in labels/comment does NOT roll back page creation — page exists, extras are best-effort
- Attachment upload not included (requires multipart/file handling, different flow)

## Testing Strategy

- **E2e tests:** Extend `McpEndpointE2ETest` with:

  - `list_spaces` returns spaces with keys and names
  - `get_page_diff` returns clean markdown diff (no `ac:macro-id` in output)
  - `get_page` uses `page` key (not `metadata`)
  - URL-based page ID resolution
  - Optimistic locking: create page, get version, update with wrong expected_version -> error
  - `append_to_page` / `prepend_to_page` add content without replacing existing
  - `convert_content` returns storage format without creating a page
  - `replace_section` updates a single section by heading
  - `create_page` with `labels` parameter applies labels
  - Tool count assertion updated to 28

- **Manual testing:** Full cycle with ChatGPT connected via MCP to verify agent experience

## Migration and Compatibility

- **Response key change** (`metadata` → `page`): This is a breaking change for any MCP client that parses the response. Since the plugin is pre-1.0 in terms of API stability and the only known consumers are ChatGPT and Claude Code, this is acceptable. Document in CHANGELOG as breaking.
- **All new parameters are optional** — existing tool calls continue to work
- **New tools** (`list_spaces`, `append_to_page`, `prepend_to_page`) are additive — no breaking changes
- **Diff format change** — markdown diff is strictly better, no opt-out needed
- **Tool count:** 23 → 28 (add `list_spaces`, `append_to_page`, `prepend_to_page`, `convert_content`, `replace_section`)

## Out of Scope

- **`get_page_by_url` as separate tool** — handled by URL extraction in `page_id` (1.4) instead
- **Date picker macros, @mentions, Jira smart links in Markdown** — would require Confluence API calls during conversion, adding latency and complexity
- **Attachment upload in batch create** — requires multipart/file handling, different flow from labels/comments
- **Specialized page creation tools** (meeting notes, procedures, status reports) — planned for v1.3+ per CLAUDE.md future architecture section

## Lessons Learned from v1.1.x Development

These findings from today's session should inform implementation:

1. **Read upstream source code deeply before implementing.** The `HTML_ALLOW_NAME_SPACE` fix was a one-line setting buried in flexmark source. Three placeholder hack attempts failed because we didn't read the source first.

2. **Flexmark configuration:** `Parser.HTML_ALLOW_NAME_SPACE = true` enables XML namespace support for inline tags. `Parser.HTML_BLOCK_TAGS` extended list enables block-level pass-through. Both are needed for Confluence XML to pass through flexmark untouched.

3. **Use `client.getRaw()` + `ResponseTransformer`** for all tools. The whitelist approach (only include fields we want) is correct. The old blacklist approach (`ResponseTrimmer`) dropped useful data.

4. **Test with real AI agents.** ChatGPT's real-world test found issues that unit tests and e2e tests missed (storage-level diff noise, response inconsistencies, missing space discovery).

5. **Tool descriptions matter enormously.** ChatGPT's initial pages didn't use callout panels because the description didn't mention them clearly enough. After improving descriptions, pages became much richer.

6. **This is a business tool, not a developer tool.** Status badges should show "Approved", "Pending Review", "Draft" — not "DONE", "IN PROGRESS", "BLOCKED". Tool descriptions should reference procedures, policies, meeting notes, reports — not just code documentation.
